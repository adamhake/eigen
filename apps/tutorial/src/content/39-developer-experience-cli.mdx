## `create-eigen` scaffolding CLI

Every popular framework ships a `create-*` CLI: `create-next-app`, `create-vite`, `create-remix`. Eigen's version is `create-eigen` — invoked via `npm create eigen`.

### How `npm create` works

When a user runs `npm create eigen`, npm:
1. Resolves the package `create-eigen`
2. Downloads and executes its `bin` entry
3. The CLI runs in the user's terminal

That's it — `npm create <name>` is shorthand for `npx create-<name>`.

### The CLI entry point

```typescript title="packages/create-eigen/src/index.ts"
#!/usr/bin/env node

import { parseArgs } from 'node:util'
import { resolve } from 'node:path'
import { mkdir, writeFile, readFile } from 'node:fs/promises'
import { downloadTemplate } from 'giget'
import prompts from 'prompts'

const { values, positionals } = parseArgs({
  allowPositionals: true,
  options: {
    typescript: { type: 'boolean', default: true },
    tailwind: { type: 'boolean' },
    template: { type: 'string', default: 'default' },
  },
})

async function main() {
  console.log('\n  create-eigen\n')

  // Project name from positional arg or prompt
  let projectName = positionals[0]
  if (!projectName) {
    const { name } = await prompts({
      type: 'text',
      name: 'name',
      message: 'Project name:',
      initial: 'my-eigen-app',
    })
    projectName = name
  }

  // Interactive prompts for options not passed as flags
  const options = await prompts([
    {
      type: values.tailwind !== undefined ? null : 'confirm',
      name: 'tailwind',
      message: 'Add Tailwind CSS?',
      initial: true,
    },
    {
      type: 'select',
      name: 'deploy',
      message: 'Deployment target:',
      choices: [
        { title: 'Node.js', value: 'node' },
        { title: 'Netlify', value: 'netlify' },
        { title: 'Cloudflare', value: 'cloudflare' },
        { title: 'None (decide later)', value: 'none' },
      ],
    },
  ])

  const config = {
    projectName,
    typescript: values.typescript ?? true,
    tailwind: values.tailwind ?? options.tailwind ?? true,
    deploy: options.deploy ?? 'node',
  }

  // Download template
  const targetDir = resolve(process.cwd(), projectName)
  await downloadTemplate(`eigen/templates/${config.deploy}`, {
    dir: targetDir,
  })

  // Apply variant overlays
  if (config.tailwind) {
    await applyOverlay(targetDir, 'tailwind')
  }

  // Post-setup
  await writeFile(
    resolve(targetDir, '.eigen.json'),
    JSON.stringify({ deploy: config.deploy }, null, 2),
  )

  console.log(`\n  Done! Created ${projectName} at ${targetDir}`)
  console.log(`\n  cd ${projectName}`)
  console.log(`  npm install`)
  console.log(`  npm run dev\n`)
}

async function applyOverlay(targetDir: string, overlay: string) {
  // Copy overlay files into the project, merging with existing
  await downloadTemplate(`eigen/overlays/${overlay}`, {
    dir: targetDir,
    force: true, // Overwrite existing files
  })
}

main().catch(console.error)
```

## Project templates

Templates are complete, runnable projects that serve as starting points. The key challenge is keeping them in sync with the framework as it evolves.

### Template structure

```
packages/create-eigen/
  templates/
    node/                    # Node.js deployment target
      src/
        pages/
          index.tsx
          about.tsx
        components/
          Layout.tsx
        app.css
      index.html
      vite.config.ts
      package.json
      tsconfig.json
    netlify/                 # Netlify deployment target
      ...same structure...
      netlify.toml
    cloudflare/              # Cloudflare deployment target
      ...same structure...
      wrangler.toml
  overlays/
    tailwind/                # Adds Tailwind to any template
      src/
        app.css              # Replaces base app.css with Tailwind imports
      vite.config.ts         # Adds tailwindcss plugin
      package.json           # Adds tailwindcss dependency (merged)
```

### Variant composition

Instead of maintaining N×M templates (every deployment target × every option), use overlays:

```
base template (node/netlify/cloudflare)
  + tailwind overlay
  = final project
```

Overlays add or replace files. The `package.json` is special — it's **merged** rather than replaced, so overlay dependencies are combined with base dependencies:

```typescript title="packages/create-eigen/src/merge.ts"
export function mergePackageJson(
  base: Record<string, unknown>,
  overlay: Record<string, unknown>,
): Record<string, unknown> {
  return {
    ...base,
    dependencies: {
      ...(base.dependencies as Record<string, string> ?? {}),
      ...(overlay.dependencies as Record<string, string> ?? {}),
    },
    devDependencies: {
      ...(base.devDependencies as Record<string, string> ?? {}),
      ...(overlay.devDependencies as Record<string, string> ?? {}),
    },
    scripts: {
      ...(base.scripts as Record<string, string> ?? {}),
      ...(overlay.scripts as Record<string, string> ?? {}),
    },
  }
}
```

## Codegen commands

Scaffolding gets you started. Codegen keeps you productive — generating routes, loaders, and components with the right conventions already wired up.

### `eigen generate route`

```bash
npx eigen generate route dashboard/settings
```

This creates:

```
src/pages/dashboard/settings.tsx    # Page component with loader
```

### AST-based code generation with `magicast`

Rather than string templates (fragile, no formatting), use `magicast` for AST-level code manipulation:

```typescript title="packages/eigen/src/cli/generate.ts"
import { parseModule, generateCode, builders } from 'magicast'
import { writeFile, mkdir } from 'node:fs/promises'
import { dirname, resolve } from 'node:path'

export async function generateRoute(routePath: string) {
  // Parse routePath: "dashboard/settings" → file + component name
  const segments = routePath.split('/')
  const componentName = segments
    .map(s => s[0].toUpperCase() + s.slice(1))
    .join('')
    + 'Page'
  const filePath = `src/pages/${routePath}.tsx`

  // Build the AST
  const mod = parseModule(`
    import { defineLoader } from 'eigen/helpers'

    export const loader = defineLoader('/${routePath}', async ({ params }) => {
      return {}
    })

    export default function ${componentName}({ data }) {
      return (
        <div>
          <h1>${segments[segments.length - 1]}</h1>
        </div>
      )
    }
  `)

  const { code } = generateCode(mod)
  const fullPath = resolve(process.cwd(), filePath)
  await mkdir(dirname(fullPath), { recursive: true })
  await writeFile(fullPath, code)

  console.log(`  Created ${filePath}`)
}
```

### `eigen generate loader`

`eigen generate loader src/pages/dashboard.tsx` adds a loader export to an existing page. It reads the file with `magicast`, checks if a loader already exists, adds the `defineLoader` import if missing, and inserts the loader export before the default export. Same AST-based approach — no fragile string templates.

### The CLI entry point

```typescript title="packages/eigen/src/cli/index.ts"
#!/usr/bin/env node

import { parseArgs } from 'node:util'
import { generateRoute, generateLoader } from './generate.js'

const { positionals } = parseArgs({
  allowPositionals: true,
  strict: false,
})

const [command, subcommand, ...args] = positionals

if (command === 'generate' || command === 'g') {
  switch (subcommand) {
    case 'route':
      await generateRoute(args[0])
      break
    case 'loader':
      await generateLoader(args[0])
      break
    default:
      console.log('Unknown generator:', subcommand)
      console.log('Available: route, loader')
  }
} else {
  console.log('eigen CLI')
  console.log('  eigen generate route <path>')
  console.log('  eigen generate loader <file>')
}
```

## Custom ESLint rules

Framework-specific linting catches mistakes that generic ESLint rules can't detect — like calling `db.query()` in a client component, or forgetting to add `"use client"` to a component that uses hooks.

### `eslint-plugin-eigen`

```typescript title="packages/eslint-plugin-eigen/src/rules/no-sync-loader.ts"
import type { Rule } from 'eslint'

/**
 * Disallows synchronous loader exports.
 * Loaders must be async to support data fetching.
 */
export const noSyncLoader: Rule.RuleModule = {
  meta: {
    type: 'problem',
    docs: {
      description: 'Loaders must be async functions',
    },
    messages: {
      syncLoader: 'Loader must be an async function. Add `async` keyword.',
    },
  },

  create(context) {
    return {
      ExportNamedDeclaration(node) {
        // Check: export const loader = (non-async function)
        if (
          node.declaration?.type === 'VariableDeclaration' &&
          node.declaration.declarations[0]?.id.type === 'Identifier' &&
          node.declaration.declarations[0].id.name === 'loader'
        ) {
          const init = node.declaration.declarations[0].init
          if (
            init?.type === 'ArrowFunctionExpression' &&
            !init.async
          ) {
            context.report({ node: init, messageId: 'syncLoader' })
          }
        }

        // Check: export function loader() (non-async)
        if (
          node.declaration?.type === 'FunctionDeclaration' &&
          node.declaration.id?.name === 'loader' &&
          !node.declaration.async
        ) {
          context.report({
            node: node.declaration,
            messageId: 'syncLoader',
          })
        }
      },
    }
  },
}
```

### Server-only boundary detection

A second rule (`server-fn-boundary`) tracks imports from `/actions/` or `.server` files, then flags direct calls in `"use client"` files — the developer should use `useServerFn()` instead. The pattern is the same: `ImportDeclaration` collects names into a `Set`, `CallExpression` checks the file directive and reports violations.

### Plugin entry point

```typescript title="packages/eslint-plugin-eigen/src/index.ts"
import { noSyncLoader } from './rules/no-sync-loader'
import { serverFnBoundary } from './rules/server-fn-boundary'

export const rules = {
  'no-sync-loader': noSyncLoader,
  'server-fn-boundary': serverFnBoundary,
}

export const configs = {
  recommended: {
    plugins: ['eigen'],
    rules: {
      'eigen/no-sync-loader': 'error',
      'eigen/server-fn-boundary': 'warn',
    },
  },
}
```

Users enable it with a single line in their ESLint config:

```javascript title="eslint.config.js"
import eigen from 'eslint-plugin-eigen'

export default [
  eigen.configs.recommended,
]
```

## TypeScript plugin

A TypeScript language service plugin runs inside the editor — it can provide autocomplete, go-to-definition, and diagnostics for framework-specific patterns.

### What it does

| Feature | Example |
|---|---|
| **Route autocomplete** | `<Link to="/` → suggests all known routes |
| **Virtual module types** | `import { routes } from 'eigen/routes'` → shows generated types |
| **Loader return type checking** | Red squiggly if `data` prop doesn't match loader return type |

### Plugin structure

```typescript title="packages/eigen/src/typescript-plugin/index.ts"
import type ts from 'typescript/lib/tsserverlibrary'

function init(modules: { typescript: typeof ts }) {
  const ts = modules.typescript

  function create(info: ts.server.PluginCreateInfo) {
    const proxy = Object.create(null) as ts.LanguageService
    const ls = info.languageService

    // Forward all methods to the original language service
    for (const k of Object.keys(ls) as Array<keyof ts.LanguageService>) {
      const x = ls[k]
      if (typeof x === 'function') {
        (proxy as any)[k] = (...args: any[]) => (x as any).apply(ls, args)
      }
    }

    // Override getCompletionsAtPosition for route autocomplete
    proxy.getCompletionsAtPosition = (fileName, position, options) => {
      const original = ls.getCompletionsAtPosition(
        fileName,
        position,
        options,
      )

      // Check if we're inside a <Link to="..." /> prop
      const sourceFile = ls.getProgram()?.getSourceFile(fileName)
      if (!sourceFile) return original

      const token = findTokenAtPosition(sourceFile, position, ts)
      if (isLinkToProp(token, ts)) {
        // Read the generated route types to suggest completions
        const routeEntries = getKnownRoutes(info.project).map(
          (route): ts.CompletionEntry => ({
            name: route,
            kind: ts.ScriptElementKind.string,
            sortText: '0', // Sort before other completions
          }),
        )

        return {
          isGlobalCompletion: false,
          isMemberCompletion: false,
          isNewIdentifierLocation: false,
          entries: [
            ...routeEntries,
            ...(original?.entries ?? []),
          ],
        }
      }

      return original
    }

    return proxy
  }

  return { create }
}

// Helper: read route paths from .eigen/routes.d.ts
function getKnownRoutes(project: ts.server.Project): string[] {
  const routesDts = project.getCurrentDirectory() + '/.eigen/routes.d.ts'
  const sourceFile = project.getLanguageService()
    .getProgram()
    ?.getSourceFile(routesDts)
  if (!sourceFile) return []
  const text = sourceFile.getFullText()
  const matches = text.matchAll(/['"](\/.+?)['"]/g)
  return [...matches].map(m => m[1])
}

// Helper: walk the AST to find the token at cursor position,
// then walk up to check if it's inside a <Link to="..."> prop.
// Implementation uses ts.forEachChild for the downward walk
// and parent references for the upward check.

export = init
```

### Registering the plugin

Users add it to their `tsconfig.json`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "plugins": [
      { "name": "eigen/typescript-plugin" }
    ]
  }
}
```

<Callout type="info" title="IDE support">
TypeScript plugins work in VS Code (with the "Use Workspace Version" TypeScript setting), WebStorm, and any editor that delegates to `tsserver`. They don't affect `tsc` on the command line — they're purely for editor intelligence.
</Callout>

### Integrating with the dev overlay

Send diagnostics to the dev overlay via the WebSocket channel established in [Part 10](/10-dev-overlay):

```typescript title="packages/eigen/src/plugins/diagnostics.ts"
import type { Plugin } from 'vite'
import { runHealthChecks } from '../diagnostics/health'

export function eigenDiagnosticsPlugin(): Plugin {
  return {
    name: 'eigen:diagnostics',

    configureServer(server) {
      // Run health checks on server start
      server.httpServer?.once('listening', async () => {
        const diagnostics = await runHealthChecks()

        if (diagnostics.length > 0) {
          // Send to dev overlay via WebSocket
          server.ws.send('eigen:diagnostics', { diagnostics })

          // Also log to terminal
          for (const d of diagnostics) {
            const prefix =
              d.level === 'error' ? '✗' :
              d.level === 'warning' ? '⚠' : 'ℹ'
            console.log(`  ${prefix} ${d.title}: ${d.message}`)
            if (d.fix) console.log(`    Fix: ${d.fix}`)
          }
        }
      })
    },
  }
}
```

The dev overlay renders diagnostics in a panel:

```tsx title="packages/eigen/src/overlay/DiagnosticsPanel.tsx"
interface DiagnosticsPanelProps {
  diagnostics: Diagnostic[]
}

export function DiagnosticsPanel({ diagnostics }: DiagnosticsPanelProps) {
  if (diagnostics.length === 0) {
    return <div className="eigen-diagnostics-ok">All checks passed</div>
  }

  return (
    <div className="eigen-diagnostics">
      {diagnostics.map((d, i) => (
        <div key={i} className={`eigen-diagnostic eigen-diagnostic-${d.level}`}>
          <strong>{d.title}</strong>
          <p>{d.message}</p>
          {d.fix && (
            <code className="eigen-diagnostic-fix">{d.fix}</code>
          )}
        </div>
      ))}
    </div>
  )
}
```

## The tooling layer in context

Here's how the DX tools relate to the runtime framework:

```
Developer writes code
  ├── create-eigen         → scaffolds the project
  ├── eigen generate       → generates routes, loaders
  ├── eslint-plugin-eigen  → catches framework-specific mistakes
  ├── TypeScript plugin    → autocomplete, go-to-definition
  └── Dev overlay          → runtime diagnostics, health checks

Developer runs dev server
  ├── Vite dev server      → HMR, transforms, virtual modules
  ├── eigen() plugin       → route discovery, SSR, server functions
  └── Diagnostics plugin   → proactive error detection
```

None of these tools change the runtime behavior. They exist to make the developer faster, catch mistakes earlier, and reduce the gap between "I want to do X" and "X is working."

## Key insight

A framework's developer experience is a force multiplier — it doesn't change what's possible, but it dramatically changes what's practical. The scaffolding CLI removes the "blank page" problem. Code generation enforces conventions without documentation. ESLint rules catch mistakes before they become bugs. The TypeScript plugin makes the type system work harder for the developer. And diagnostics surface problems before the developer even knows to look.

The common thread: every DX tool is built from the same information the framework already has — the route tree, the module graph, the type definitions. The runtime framework *knows* your routes, your loaders, your server functions. The DX layer just makes that knowledge accessible to the developer in the right place at the right time.
