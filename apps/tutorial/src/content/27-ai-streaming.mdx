## Why AI streaming is a framework concern

When an application calls an LLM, the response arrives as a stream of tokens over seconds. If the server function from Part 15 waits for the full response before returning, the user stares at a loading spinner for 3-10 seconds. Streaming the response token-by-token — showing text as it generates — transforms the UX from "waiting" to "watching it think."

This isn't just a `fetch` with `ReadableStream`. AI responses are *structured*: they contain text chunks, tool call invocations (the model wants to run a function), tool results, and metadata. The framework needs to serialize this structure across the server/client boundary and provide typed React hooks to consume it.

[TanStack AI](https://tanstack.com/ai) launched in alpha in late 2025 as a framework-agnostic library for exactly this: typed streaming of AI model responses through server functions, with React hooks for consuming them. It builds on the [Vercel AI SDK](https://sdk.vercel.ai) (`ai` package), which provides the low-level primitives for calling LLMs (`streamText`, `streamObject`) and structuring their responses.

## Extending server functions for streaming

In Part 15, server functions return `Promise<T>` — a single value. For AI, we need `ReadableStream<T>` — an async iterable of typed chunks:

```typescript title="packages/eigen/server.ts"
export function createStreamingServerFn<TChunk>(
  fn: () => AsyncGenerator<TChunk>,
): () => ReadableStream<TChunk> {
  return fn as any // Server: runs the generator. Client: transform replaces with SSE fetch.
}
```

### The streaming server function convention

```typescript title="src/server/chat.ts"
import { createStreamingServerFn } from 'eigen/server'
import { streamText } from 'ai'  // Vercel AI SDK or similar

export const chat = createStreamingServerFn(async function* (
  messages: Array<{ role: string; content: string }>,
) {
  const result = streamText({
    model: openai('gpt-4o'),
    messages,
    tools: {
      getWeather: {
        description: 'Get weather for a location',
        parameters: z.object({ location: z.string() }),
        execute: async ({ location }) => {
          const weather = await fetchWeather(location)
          return weather
        },
      },
    },
  })

  // Yield structured chunks as they arrive
  for await (const chunk of result.fullStream) {
    yield chunk  // { type: 'text-delta', textDelta: '...' }
                 // { type: 'tool-call', toolName: 'getWeather', args: {...} }
                 // { type: 'tool-result', toolName: 'getWeather', result: {...} }
  }
})
```

### The transform: streaming RPC

The `transform` hook from Part 15 generates a different client stub for streaming functions — one that returns a `ReadableStream` from a server-sent events (SSE) connection:

```typescript
// Client stub generated by the transform hook:
export const chat = (messages) => {
  const response = await fetch('/_eigen/fn/stream/b4e2a1', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(messages),
  })

  // Return a ReadableStream that decodes SSE events into typed chunks
  return response.body
    .pipeThrough(new TextDecoderStream())
    .pipeThrough(new SSEDecoderStream())
}
```

### The dev server endpoint

```typescript
// In configureServer — streaming function handler
server.middlewares.use(async (req, res, next) => {
  if (!req.url?.startsWith('/_eigen/fn/stream/')) return next()

  const fnId = req.url.replace('/_eigen/fn/stream/', '').split('?')[0]
  const registration = fnRegistry.get(fnId)
  if (!registration) { res.writeHead(404); res.end(); return }

  // Parse request body
  const chunks: Buffer[] = []
  for await (const chunk of req) chunks.push(chunk as Buffer)
  const args = JSON.parse(Buffer.concat(chunks).toString())

  // Set up SSE headers
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
  })

  // Load and call the streaming function
  const mod = await server.ssrLoadModule(registration.file)
  const fn = mod[registration.exportName]
  const generator = fn(args)

  // Stream each yielded chunk as an SSE event
  for await (const chunk of generator) {
    res.write(`data: ${JSON.stringify(chunk)}\n\n`)
  }

  res.write('data: [DONE]\n\n')
  res.end()
})
```

## Typed React hooks for AI streams

The framework provides hooks that consume the typed stream and manage UI state:

```typescript title="packages/eigen/ai-hooks.ts"
import { useState, useCallback } from 'react'

interface StreamChunk {
  type: 'text-delta' | 'tool-call' | 'tool-result' | 'finish'
  textDelta?: string
  toolName?: string
  args?: unknown
  result?: unknown
}

interface UseStreamResult {
  messages: Array<{ role: string; content: string }>
  isStreaming: boolean
  send: (content: string) => void
  activeToolCalls: Array<{ name: string; args: unknown }>
}

export function useChat(
  serverFn: (messages: Array<{ role: string; content: string }>) => ReadableStream<StreamChunk>,
): UseStreamResult {
  const [messages, setMessages] = useState<Array<{ role: string; content: string }>>([])
  const [isStreaming, setIsStreaming] = useState(false)
  const [activeToolCalls, setActiveToolCalls] = useState<Array<{ name: string; args: unknown }>>([])

  const send = useCallback(async (content: string) => {
    const userMessage = { role: 'user' as const, content }
    const updatedMessages = [...messages, userMessage]
    setMessages(updatedMessages)
    setIsStreaming(true)

    let assistantContent = ''

    try {
      const stream = await serverFn(updatedMessages)
      const reader = stream.getReader()

      while (true) {
        const { done, value } = await reader.read()
        if (done) break

        const chunk = value as StreamChunk

        switch (chunk.type) {
          case 'text-delta':
            assistantContent += chunk.textDelta ?? ''
            setMessages([...updatedMessages, { role: 'assistant', content: assistantContent }])
            break

          case 'tool-call':
            setActiveToolCalls(prev => [...prev, { name: chunk.toolName!, args: chunk.args }])
            break

          case 'tool-result':
            setActiveToolCalls(prev => prev.filter(t => t.name !== chunk.toolName))
            break
        }
      }
    } finally {
      setIsStreaming(false)
    }
  }, [messages, serverFn])

  return { messages, isStreaming, send, activeToolCalls }
}
```

### Usage in a component

```tsx title="src/pages/chat.tsx"
'use client'

import { useChat } from 'eigen/ai-hooks'
import { chat } from '../server/chat'

export default function ChatPage() {
  const { messages, isStreaming, send, activeToolCalls } = useChat(chat)

  return (
    <div>
      {messages.map((m, i) => (
        <div key={i} className={m.role}>
          {m.content}
        </div>
      ))}

      {activeToolCalls.map(tc => (
        <div key={tc.name} className="tool-call">
          🔧 Calling {tc.name}({JSON.stringify(tc.args)})...
        </div>
      ))}

      {isStreaming && <div className="typing-indicator">●●●</div>}

      <input
        onKeyDown={e => {
          if (e.key === 'Enter') {
            send(e.currentTarget.value)
            e.currentTarget.value = ''
          }
        }}
        placeholder="Type a message..."
      />
    </div>
  )
}
```

## Typed tool calls

The most interesting type safety challenge: tool calls are defined on the server (in the streaming function), but the client needs to know about them for rendering. The framework can generate types from the tool definitions:

```typescript
// The server defines tools with zod schemas
const tools = {
  getWeather: {
    parameters: z.object({ location: z.string() }),
    execute: async ({ location }) => fetchWeather(location),
  },
  searchProducts: {
    parameters: z.object({ query: z.string(), maxResults: z.number() }),
    execute: async ({ query, maxResults }) => searchDB(query, maxResults),
  },
} as const

// The framework infers:
type ToolCallEvent =
  | { type: 'tool-call'; toolName: 'getWeather'; args: { location: string } }
  | { type: 'tool-call'; toolName: 'searchProducts'; args: { query: string; maxResults: number } }
```

This is the same pattern as server functions (Part 15): TypeScript checks the source file where tools are defined, the `transform` hook generates the client stub, and the types survive because the type checker operates on source, not transformed output.

## Structured object streaming

Beyond chat, AI models can generate structured data (JSON objects) progressively. The Vercel AI SDK's `streamObject` returns partial objects as they're generated:

```typescript
export const generateReport = createStreamingServerFn(async function* (topic: string) {
  const result = streamObject({
    model: openai('gpt-4o'),
    schema: z.object({
      title: z.string(),
      sections: z.array(z.object({
        heading: z.string(),
        content: z.string(),
      })),
      summary: z.string(),
    }),
  })

  for await (const partial of result.partialObjectStream) {
    yield partial  // Typed as DeepPartial<ReportSchema>
  }
})
```

The client receives a progressively-completing object — first `{ title: "..." }`, then `{ title: "...", sections: [{ heading: "..." }] }`, and so on. The component renders whatever's available, filling in as more arrives.

## What to observe

1. **Watch the Network tab** during a chat interaction. You'll see an SSE connection that receives chunks progressively — no full-page requests.

2. **Type a message and watch the response build** character by character. The text streams in via the `text-delta` chunks.

3. **Trigger a tool call** (ask about weather). The `tool-call` chunk appears in the stream, the UI shows a "calling tool" indicator, then the `tool-result` chunk arrives with the data.

4. **Check the types** — hover over the `activeToolCalls` array. TypeScript knows the possible tool names and argument shapes.

## Key insight

AI streaming extends the server function model from "request/response" to "request/stream." The same `transform` hook generates the client stub, but instead of returning `Promise<T>` it returns `ReadableStream<T>`. The SSE transport replaces the JSON response. The type safety story is unchanged — TypeScript checks the source, the transform generates the stub, and the types survive.

This pattern isn't AI-specific. Any use case that produces structured data over time — real-time analytics, file processing progress, multi-step workflows — benefits from the same streaming server function primitive. AI is just the most compelling current application.
