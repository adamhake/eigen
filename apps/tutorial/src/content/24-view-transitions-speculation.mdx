## View Transitions: CSS-driven page animations

The View Transitions API lets the browser capture a "before" snapshot of the page, update the DOM, capture an "after" snapshot, and animate between them — all driven by CSS. The default is a crossfade, but you can define per-element transitions for shared elements (like a card expanding into a detail page).

### Same-document transitions (SPA)

In our Navigation API intercept handler, wrap the DOM update in `startViewTransition`:

```typescript
event.intercept({
  async handler() {
    const res = await fetch(`/_eigen/data?path=${encodeURIComponent(url.pathname)}`, {
      signal: event.signal,
    })
    const newData = await res.json()

    // Wrap the state update in a view transition
    const transition = document.startViewTransition(() => {
      // This callback runs synchronously — React flushes the update
      flushSync(() => {
        setPathname(url.pathname)
        setData(newData)
      })
    })

    // Optionally wait for the animation to complete
    await transition.finished
  },
})
```

The browser automatically creates `::view-transition-old(root)` and `::view-transition-new(root)` pseudo-elements and crossfades between them. You control the animation entirely in CSS:

```css
/* Default crossfade — applied automatically */
::view-transition-old(root) {
  animation: fade-out 200ms ease;
}
::view-transition-new(root) {
  animation: fade-in 200ms ease;
}

/* Slide animation for navigations */
@keyframes slide-out {
  to { transform: translateX(-100%); }
}
@keyframes slide-in {
  from { transform: translateX(100%); }
}

::view-transition-old(root) {
  animation: slide-out 300ms ease;
}
::view-transition-new(root) {
  animation: slide-in 300ms ease;
}
```

### Named transitions for shared elements

The real power is per-element transitions. Give elements matching `view-transition-name` values, and the browser animates them individually:

```tsx
// On the list page
function PostCard({ post }: { post: Post }) {
  return (
    <article style={{ viewTransitionName: `post-${post.id}` }}>
      <h2>{post.title}</h2>
      <img src={post.thumbnail} alt="" />
    </article>
  )
}

// On the detail page
function PostDetail({ data }: { data: Post }) {
  return (
    <article style={{ viewTransitionName: `post-${data.id}` }}>
      <h1>{data.title}</h1>
      <img src={data.image} alt="" />
    </article>
  )
}
```

The browser detects that `post-42` exists in both the old and new snapshots, and animates the element smoothly from its list position/size to its detail position/size. No JavaScript animation library needed.

### Transition types for directional animations

The Navigation API provides navigation direction, which we can use to set transition types:

```typescript
event.intercept({
  async handler() {
    // Determine direction based on history position
    const isBack = event.navigationType === 'traverse'
      && (event.destination.index ?? 0) < (navigation.currentEntry?.index ?? 0)

    // Set the transition type — CSS uses this to choose the animation
    const transition = document.startViewTransition({
      update: () => flushSync(() => { setPathname(url.pathname); setData(newData) }),
      types: [isBack ? 'slide-backward' : 'slide-forward'],
    })
  },
})
```

```css
/* Forward navigation: slide left */
@view-transition(slide-forward) {
  ::view-transition-old(root) { animation: slide-out-left 300ms; }
  ::view-transition-new(root) { animation: slide-in-right 300ms; }
}

/* Back navigation: slide right */
@view-transition(slide-backward) {
  ::view-transition-old(root) { animation: slide-in-left 300ms reverse; }
  ::view-transition-new(root) { animation: slide-out-right 300ms reverse; }
}
```

## Cross-document view transitions (MPA)

For pages that don't use client-side routing (or before JavaScript loads), cross-document view transitions animate between full page loads. Enable them with a CSS at-rule:

```css
@view-transition {
  navigation: auto;
}
```

That single line gives every same-origin navigation a crossfade animation. This is the progressive enhancement story: the page works without JavaScript, cross-document transitions provide the base animation, and when the Navigation API intercept kicks in, same-document transitions provide the richer experience.

### Generating the CSS via `transformIndexHtml`

The framework plugin can inject the view transition CSS automatically:

```typescript
// In the eigen plugin
transformIndexHtml(html: string) {
  return html.replace(
    '</head>',
    `<style>
      @view-transition { navigation: auto; }
      ::view-transition-old(root) { animation-duration: 200ms; }
      ::view-transition-new(root) { animation-duration: 200ms; }
    </style></head>`,
  )
}
```

## Speculation Rules: browser-native prerendering

The Speculation Rules API tells the browser to prefetch or prerender pages *before* the user navigates to them. Combined with view transitions, this creates zero-latency animated navigations — the page is already rendered in a hidden tab when the user clicks.

### Static rules

```html
<script type="speculationrules">
{
  "prerender": [
    {
      "where": { "href_matches": "/*" },
      "eagerness": "moderate"
    }
  ]
}
</script>
```

Eagerness levels: `immediate` (prerender on page load), `eager` (prerender on hover intent), `moderate` (prerender on hover/pointer down), `conservative` (prerender on click).

### Generating rules from the route tree

The framework plugin knows all routes. It can generate targeted speculation rules:

```typescript
// In the eigen plugin — generate speculation rules
transformIndexHtml(html: string, ctx) {
  const routes = discoverRoutes(pagesDir)

  // Only prerender static routes (no dynamic segments) and popular pages
  const staticRoutes = routes
    .filter(r => !r.path.includes(':'))
    .map(r => r.path)

  const rules = {
    prerender: [
      {
        urls: staticRoutes,
        eagerness: 'moderate' as const,
      },
    ],
    prefetch: [
      {
        // Dynamic routes: prefetch (but don't prerender) on hover
        where: { href_matches: '/*' },
        eagerness: 'moderate' as const,
      },
    ],
  }

  return html.replace(
    '</head>',
    `<script type="speculationrules">${JSON.stringify(rules)}</script></head>`,
  )
}
```

### Dynamic rules based on the current page

More sophisticated: the plugin generates per-page rules based on the route tree. The `/dashboard` page prerenders its child routes because those are the most likely next navigations:

```typescript
// Generated for /dashboard
{
  "prerender": [
    {
      "urls": ["/dashboard/settings", "/dashboard/analytics"],
      "eagerness": "eager"
    }
  ]
}
```

This can be generated by the virtual module — each page's speculation rules are based on its position in the route tree and its outgoing links.

## The combined effect

When View Transitions, Speculation Rules, and the Navigation API work together:

1. **User hovers over a link.** Speculation Rules trigger prerendering of the target page.
2. **User clicks.** The Navigation API's `navigate` event fires.
3. **The intercept handler detects** the page is already prerendered in a hidden tab.
4. **View Transition captures** the old page snapshot.
5. **The prerendered page activates** (near-instant — it's already rendered).
6. **View Transition animates** from old to new with a CSS-driven crossfade/slide.

Total perceived latency: the animation duration (~200ms). Data fetching, rendering, and JavaScript execution happened during the hover — invisible to the user.

This is the experience that native apps provide. The browser is now capable of it without any framework JavaScript for the prerendering step.

## Respecting user preferences

Always respect `prefers-reduced-motion`:

```css
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}
```

And the speculation rules should be conditional on network quality:

```typescript
// Don't prerender on slow connections
const shouldPrerender =
  !navigator.connection?.saveData &&
  (navigator.connection?.effectiveType ?? '4g') !== '2g'
```

## What to observe

1. **Navigate between pages** and watch the crossfade animation. It's driven entirely by CSS — no JavaScript animation library.

2. **Add `view-transition-name` to shared elements** (like a card that expands into a detail page). Watch the browser animate the element's position and size across the navigation.

3. **Open Chrome DevTools → Application → Speculative loads.** You'll see which pages are being prerendered and their status.

4. **Throttle the network to "Slow 3G"** and navigate. Without speculation rules, you see the loading state. With speculation rules on moderate eagerness, the page loads instantly if you hovered for >200ms before clicking.

5. **Disable JavaScript entirely.** Cross-document view transitions (the CSS `@view-transition` rule) still work — the animation happens between full page loads.

## Key insight

These three browser APIs (Navigation, View Transitions, Speculation Rules) represent a fundamental shift in what the *platform* provides vs. what *frameworks* must implement. Client-side routing, page transition animations, and predictive preloading have been framework concerns for a decade. They're becoming browser primitives.

For a Vite framework plugin, the role evolves from "implement routing" to "generate metadata that feeds browser APIs." The plugin generates speculation rules from the route tree (via `transformIndexHtml`), provides type-safe navigation (via the virtual module's `RoutePaths` type), and lets the browser handle the rest. Less JavaScript in the bundle. Better performance. Progressive enhancement by default.

Astro pioneered this approach with its built-in View Transition support. The pattern is now available to any framework — including Eigen.
