*In this installment, we explore how a framework connects to external services — databases, analytics, error tracking, and feature flags — through the plugin infrastructure. Rather than treating each service as an isolated npm import, we show how the framework plugin can generate configuration, inject scripts, instrument server functions, and provide typed helpers that make integration seamless.*
---

## The integration pattern

External services interact with the framework at multiple layers:

| Layer | Example | Hook / mechanism |
|---|---|---|
| **Build time** | Validate env vars exist | `configResolved` |
| **HTML injection** | PostHog analytics script | `transformIndexHtml` |
| **Middleware** | Database connection per request | Framework middleware (Part 14) |
| **Loaders** | Query Neon in data loaders | Direct import in server code |
| **Server functions** | Track events on mutations | Instrumentation in `transform` or middleware |
| **Client components** | PostHog event tracking | Client-side SDK initialization |
| **Error handling** | Report errors to PostHog | Error boundary integration |
| **Adapter output** | Connection string in deploy config | Adapter plugin (Part 16) |

A well-designed integration plugin touches many of these layers through a single configuration:

```typescript title="vite.config.ts"
import eigen from 'eigen/plugin'
import { neonIntegration } from 'eigen/integrations/neon'
import { posthogIntegration } from 'eigen/integrations/posthog'

export default defineConfig({
  plugins: [
    eigen(),
    neonIntegration(),     // Database
    posthogIntegration(),  // Analytics + error tracking
    react(),
  ],
})
```

## Neon integration

### The serverless database challenge

Neon is a serverless Postgres provider. In serverless environments (Netlify Functions, Cloudflare Workers, Vercel Edge), traditional connection pooling doesn't work — each function invocation is isolated. Neon's `@neondatabase/serverless` driver uses HTTP or WebSocket connections that work in these environments.

The framework integration handles connection management, so developers just write queries:

```typescript title="packages/eigen/integrations/neon.ts"
import type { Plugin, ResolvedConfig } from 'vite'

export function neonIntegration(): Plugin {
  return {
    name: 'eigen-integration-neon',

    configResolved(config: ResolvedConfig) {
      // Validate that the connection string is available
      const dbUrl = process.env.DATABASE_URL
      if (!dbUrl) {
        console.warn(
          '⚠ DATABASE_URL not set. Neon integration requires a connection string.\n' +
          '  Set it in .env.local: DATABASE_URL=postgresql://...',
        )
      }

      // Validate it looks like a Neon URL
      if (dbUrl && !dbUrl.includes('neon.tech') && !dbUrl.includes('localhost')) {
        console.warn(
          '⚠ DATABASE_URL does not appear to be a Neon connection string.',
        )
      }
    },

    // Generate a typed database helper that handles connection lifecycle
    resolveId(id) {
      if (id === 'eigen/db') return '\0eigen/db'
    },

    load(id) {
      if (id !== '\0eigen/db') return

      const isEdge = this.environment?.name === 'edge'
      const isSSR = this.environment?.name === 'ssr' || this.environment?.name === 'rsc'

      if (!isSSR && !isEdge) {
        // Client environment — db module should not be importable
        return `
          throw new Error(
            'Cannot import database module in client code. ' +
            'Use a server function or loader instead.'
          )
        `
      }

      if (isEdge) {
        // Edge: use Neon's HTTP driver (no WebSocket in all edge runtimes)
        return `
          import { neon } from '@neondatabase/serverless'

          const sql = neon(process.env.DATABASE_URL)

          export { sql }
          export async function query(text, params) {
            return sql(text, params)
          }
        `
      }

      // Node SSR: use Neon's WebSocket driver with connection pooling
      return `
        import { Pool } from '@neondatabase/serverless'
        import ws from 'ws'
        import { neonConfig } from '@neondatabase/serverless'

        // Configure WebSocket for Node.js
        neonConfig.webSocketConstructor = ws

        let pool = null

        function getPool() {
          if (!pool) {
            pool = new Pool({
              connectionString: process.env.DATABASE_URL,
              max: 10,
            })
          }
          return pool
        }

        export async function query(text, params) {
          const client = await getPool().connect()
          try {
            return await client.query(text, params)
          } finally {
            client.release()
          }
        }

        export const sql = getPool()
      `
    },
  }
}
```

### Type declarations for the database module

```typescript title="node_modules/.eigen/eigen-db.d.ts"
declare module 'eigen/db' {
  import type { QueryResult } from '@neondatabase/serverless'

  /** Execute a parameterized SQL query */
  export function query<T extends Record<string, unknown> = Record<string, unknown>>(
    text: string,
    params?: unknown[],
  ): Promise<QueryResult<T>>

  /** The connection pool (Node) or HTTP client (Edge) */
  export const sql: unknown
}
```

### Usage in a loader

```tsx title="src/pages/posts/[id].tsx"
import { query } from 'eigen/db'
import { defineLoader } from 'eigen/helpers'

export const loader = defineLoader('/posts/:id', async ({ params }) => {
  const result = await query<{ id: string; title: string; body: string }>(
    'SELECT id, title, body FROM posts WHERE id = $1',
    [params.id],
  )
  return result.rows[0]
})
```

The import `eigen/db` is a virtual module that the Neon integration plugin provides. It generates different code per environment — WebSocket-pooled connections for Node, HTTP connections for edge. And the client environment gets a hard error if you try to import it, preventing accidental database access in browser code.

### Drizzle ORM integration

For a richer DX, the integration can expose a Drizzle client:

## PostHog integration

PostHog serves multiple roles: product analytics, session replay, feature flags, error tracking. The integration plugin handles all of them:

### Usage in middleware and loaders

```typescript title="src/middleware/analytics.ts"
import { defineMiddleware } from 'eigen/middleware'
import { trackEvent } from 'eigen/analytics'

export const analyticsMiddleware = defineMiddleware(async (ctx) => {
  // Track page views server-side (more reliable than client-side)
  const userId = ctx.user?.id ?? 'anonymous'
  trackEvent('$pageview', {
    $current_url: ctx.pathname,
    $referrer: ctx.request.headers.get('referer') ?? '',
  }, userId)

  return {}
})
```

```typescript
// In a server function — track mutations
export const createPost = createServerFn(async (data, ctx) => {
  const post = await query('INSERT INTO posts ...', [data.title])

  trackEvent('post_created', {
    post_id: post.id,
    title: data.title,
  }, ctx.user.id)

  return post
})
```

## The integration plugin pattern

Both Neon and PostHog integrations follow the same pattern:

1. **`configResolved`** — Validate environment variables exist and are well-formed
2. **`resolveId` / `load`** — Generate virtual modules with environment-appropriate code (client vs. SSR vs. edge)
3. **`transformIndexHtml`** — Inject client-side scripts (analytics snippets)
4. **Type declarations** — Generate `.d.ts` files for the virtual modules
5. **Middleware** — Provide pre-built middleware for common patterns (connection management, event tracking)

This pattern is extensible to any service: Sentry for error tracking, LaunchDarkly for feature flags, Stripe for payments, Resend for email. Each integration is a Vite plugin that wires the service into the framework at the appropriate layers.

---

## Environment-aware code generation

The most interesting aspect is how the virtual module generates different code per environment:

| Module | Client | SSR (Node) | Edge |
|---|---|---|---|
| `eigen/db` | Error (can't use DB in browser) | WebSocket pool | HTTP driver |
| `eigen/analytics` | `posthog.capture()` (browser SDK) | PostHog Node SDK | PostHog Node SDK (lite) |

This is the virtual module pattern from Part 2, applied to infrastructure. The developer writes `import { query } from 'eigen/db'` and the framework provides the right implementation for the target runtime.

---

## Key insight

Platform integrations are where the framework's architecture pays off most visibly. Without the framework, connecting Neon to a serverless deployment requires manually configuring connection pooling, handling the WebSocket driver setup, preventing client-side imports, and managing environment variables. With the integration plugin, it's `neonIntegration()` in the config and `import { query } from 'eigen/db'` in your code.

The virtual module pattern ensures the right driver is used per environment. The `configResolved` hook validates configuration at startup. The `transformIndexHtml` hook injects client scripts. And the middleware layer provides request-scoped concerns like analytics tracking.

This is the "batteries-included" experience that makes frameworks valuable beyond their rendering capabilities. The rendering pipeline (Parts 0–20) is the foundation; the integration layer is what makes it practical for real applications.
